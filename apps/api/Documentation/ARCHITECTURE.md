ChessBox backend — status as of July 3, 2026
Monorepo restructure is complete: apps/desktop (Electron + React + TS, using tRPC client + React Query + superjson) and apps/api (Fastify + tRPC + Prisma, planned) live under one repo root with npm workspaces (apps/*, packages/* — packages/shared intended for zod schemas and blacklist.json shared between both apps, not yet built). Stack: Fastify, tRPC, PostgreSQL, Prisma ORM, hash-wasm (chosen over argon2 for ARM/Raspberry Pi compatibility), SendGrid, JWT, Redis, deployed eventually to AWS via Docker, reached from the desktop app over a Cloudflare Tunnel.
Postgres is running locally via Docker (docker-compose.yml at repo root, postgres:16 image, db/user/password all chessbox/chessbox/devpassword, named volume pgdata for persistence). apps/api/.env holds DATABASE_URL plus a Lichess API token. Prisma is initialized (apps/api/prisma/schema.prisma, prisma.config.ts with dotenv/config imported) with one model so far — User (id, username, email, passwordHash, createdAt) — and the initial migration (20260703213804_init) has been applied successfully. Confirmed working end-to-end: container up, psql connects, Prisma Client can reach the DB, prisma migrate dev runs clean.
One environment gotcha worth remembering: this machine also has a native Postgres 18 Windows service (postgresql-x64-18) that was fighting Docker for port 5432. It's been stopped and set to Manual startup — if prisma migrate or the API ever throws P1000 authentication errors again after a reboot, check Get-Service postgresql-x64-18 and netstat -ano | findstr :5432 first, since it may have restarted.
Not started yet: the actual Fastify server (apps/api/src/index.ts), tRPC router/context wiring, the auth.service.ts (signup/login using hash-wasm + blacklist check), and connecting the desktop logIn.tsx component to any of it. Also flagged but not yet cleaned up: apps/desktop/package.json still has some backend-looking deps (apify-client, bad-words, argon2id) left over from before the split that should probably move to apps/api or get removed.
Next step when resuming: build the Fastify server with a basic health-check route to confirm it can boot and reach Postgres via Prisma Client, before layering tRPC auth procedures on top.


ChessBox backend — status as of July 10, 2026
Backend status unchanged from July 3 (Postgres in Docker via docker-compose.yml, postgres:16, named volume pgdata; Prisma initialized with User model and initial migration applied and confirmed working). Today's session was conceptual, not code: clarified why Docker is being used at all, since this is the first time using it. Key resolved confusions — Docker Compose (already in use) runs pre-built images pulled from Docker Hub (e.g. postgres:16); a Dockerfile is a different thing entirely — a recipe for building a custom image from your own source code, and is only needed when packaging the Fastify API for deployment (AWS), not for local dev. Decided not to write a Dockerfile yet, since apps/api/src/ doesn't exist yet and there's nothing to containerize — writing one now would just mean rewriting it repeatedly against a moving target. Confirmed the intended local-dev pattern going forward: Postgres and (later) Redis run in Docker via docker-compose.yml; the Fastify API itself runs natively during dev (tsx watch or similar, for fast hot-reload/debugging) and only gets a Dockerfile once actual AWS deployment is in scope. Electron/React desktop app never gets containerized — it's a native client app that talks to the API remotely via the Cloudflare Tunnel.
Next step when resuming: exactly as before — build apps/api/src/index.ts, a minimal Fastify server with a /health route that does a prisma.$queryRaw (or similar) to confirm it can reach Postgres through Prisma Client end-to-end.
